## Design Decisions（SPEC 準拠の確定事項）

実装時に確定し、以降のフェーズもこれに従う:

1. **`flovia` extension は flat**（SPEC「Personalized HTTP 402 Response」L740-783, canonical shape L497-526 準拠）。
   ```json
   "flovia": {
     "type": "verified_user_discount",
     "base_price": "0.05",
     "final_price": "0.025",
     "currency": "USDC",
     "policy": "verified_user_discount",
     "reason_codes": ["privy_agent_authorized", "verified_privy_user"],
     "unlock": { "type": "link_account", "condition": "...", "target_final_price": "0.025" }
   }
   ```
   - `buyer` は **quote response（Flovia API 内部）にだけ**含め、`flovia` には出さない（merchant 非公開原則 / `agent_score` 漏洩防止）。
   - `request_id` / `quote_id` は `flovia` に入れず、**x402 準拠の `accepts[0].extra`** に格納する（agent はそこから取得）。
2. **identity は merchant から送らせない**。Flovia が wallet → user store で解決する（SPEC L350, Authorization Model L528-543）。
3. **`policy` は pricing policy のみ**。next-offer 型（starter_upsell / loyalty_bundle）は別 schema（`NextOfferTypeSchema`）として扱う。

## Phase 1: Canonical Shared Contract — ✅ done

目的: `packages/shared` を最新 SPEC の source of truth にする。

実装済み:

- `SegmentSchema`: `anonymous_wallet` / `low_assurance_privy_user` / `verified_privy_user`
  - repeat buyer は segment enum ではなく payment history（`merchantPaidCalls`）で扱う（Phase 5）。
- `PolicySchema`: `base_price` / `base_price_until_verified` / `verified_user_discount`
- `OfferTypeSchema`: `base_price` / `unlockable_discount` / `verified_user_discount`
- `IdentityConfidenceSchema`: `anonymous` / `wallet_only` / `verified_contact` / `verified_social` / `strong_auth`
- `LinkedAccountTypeSchema`: `email` / `farcaster` / `github` / `passkey` / `mfa`
- `UnlockSchema`: `{ type:"link_account", condition, target_final_price }`
- `OfferSchema`（canonical）: `{ type, base_price, final_price, currency, policy, unlock? }`
- `BuyerSchema`: `{ wallet, source, segment }`（`agent_score` は含めない）
- `OfferQuoteResponseSchema`: `{ quote_id, buyer, offer, reason_codes }`
- `FloviaExtensionSchema`: `OfferSchema` + `reason_codes`（**flat**。`buyer` / `request_id` / `quote_id` を含めない）
- money helpers: `toAtomic` / `fromAtomic` / `applyRate`（USDC 6 decimals, BigInt）

Phase 5 で追加済み:

- ✅ `NextOfferTypeSchema`（`starter_upsell` / `loyalty_bundle`）と `FloviaNextOfferSchema`。

注意点:

- Flovia API payload と `flovia` extension は `snake_case`。
- `basePrice` / `finalPrice` / `reasonCodes` / `alternativeEndpoints` / `premiumUpsell` 等の camelCase wire field は廃止済み。
- UI 表示用の camelCase 変換は frontend 内だけに閉じ込める。

検証: ✅ `bun run typecheck` / API quote response・merchant 402 を Zod parse 可能。

## Phase 2: Identity Simulation Model — ✅ done

目的: real Privy の前に、wallet-only と verified を backend が区別できるようにする。

実装済み（SPEC 準拠版）:

- **`OfferQuoteRequestSchema.metadata` には identity を入れない**（`task_category` / `agent_declared_budget` のみ）。identity は wallet から server-side 解決する。
  - ※ 旧プランの `identity_confidence` / `linked_account_types` / `privy_did` を metadata に送る案は privacy 境界違反のため破棄。
- `packages/db` に simulation user store を追加:
  - wallet address / optional privy DID / identity confidence / linked account types / agent authorization state / createdAt
  - `getUserByWallet` / `upsertUser` / `linkAccount`（link 種別 → identity confidence を昇格）
- `apps/api` に simulation endpoints（dev only / 本番ではゲート or 削除）:
  - `POST /v1/dev/users`（Privy login + agent authorization の stand-in。`authorized` フィールドで認可状態を設定）
  - `POST /v1/dev/users/:wallet/link`（Privy での account linking の stand-in）
  - ⏳ optional: `GET /v1/dev/users/:wallet`、認可だけを切る `.../authorize-agent` は必要になれば追加。
- merchant quote flow は wallet から local simulation user を解決（`apps/api` の `/v1/offers/quote`）。

Demo states:

- anonymous wallet: Flovia user 無し → base price
- wallet-only Privy user: `identity_confidence = wallet_only` + authorized
- verified Privy user: `verified_contact` / `verified_social` + authorized
- strong auth user: `strong_auth` + authorized

検証: ✅

- wallet-only → `unlockable_discount`, final = base。
- verified → `verified_user_discount`, final = base × 0.5。
- anonymous → base price, unlock prompt 無し。
- 認可なし（`authorized=false`）→ rule 1 で base へフォールバック。

## Phase 3: Offer Engine Alignment — ✅ done

目的: Pattern 1 を core path として正しく実装する。

実装済み:

- `computeOffer` は identity confidence based。
- Pricing rules:
  - anonymous: `type=base_price`, `policy=base_price`, `final_price=base_price`
  - wallet-only + authorized: `type=unlockable_discount`, `policy=base_price_until_verified`, `final_price=base_price`, `unlock.target_final_price=base_price × 0.5`
  - verified contact/social + authorized: `type=verified_user_discount`, `policy=verified_user_discount`, `final_price=base_price × 0.5`
  - strong auth + authorized: verified user と同価格、reason code は `strong_auth_privy_user`
- `reason_codes`: `anonymous_wallet` / `low_identity_confidence` / `privy_agent_authorized` / `verified_privy_user` / `strong_auth_privy_user`
- repeat buyer / bundle logic は current-request pricing から外し、Phase 5 の `flovia_next_offer` に回す。
- `computeAgentScore` は **future-only / MVP pricing では未使用**（内部・dashboard 用に残置、merchant には出さない）。

Money handling（実装済み helper, in `packages/shared`）:

- `toAtomic("0.05") -> "50000"`
- `fromAtomic("25000") -> "0.025"`
- `applyRate("0.05", 1, 2) -> "0.025"`（BigInt 演算）
- ⏳ 必要なら bps API（`multiplyBps`）を追加してもよい。

検証: ✅

- `0.05 × 0.5 = 0.025`、端数 `0.07 × 0.5 = 0.035`（float ドリフト無し）。
- ⏳ `0.01` / `0.025` / `0.05` / `0.08` の roundtrip test を Phase 7 で追加。

## Phase 4: SDK And Merchant 402 Response — ✅ done

目的: merchant response を SPEC の canonical 402 に揃える。

実装済み:

- `flovia` は flat canonical shape（Design Decisions #1）。
- `accepts[0]` は x402-shaped:
  - `scheme` / `network` / `maxAmountRequired`（atomic units）/ `asset`（network 別 USDC address）/ `payTo`
  - `extra: { quote_id, request_id, display_price, currency, simulation: true }`
- `config` に asset address を追加（`usdcAssetAddresses.base` / `["base-sepolia"]`）。
- `X-PAYMENT` parse は simulation 用として明示。
- demo-agent と（残）README を新 shape に合わせる。

追加実装済み:

- ✅ SDK option に `paymentMode: "simulation" | "x402"` を追加。`simulation` は `extra.simulation=true`、`x402` は simulation marker を外した facilitator-facing shape を返す。

Compatibility policy:

- 外部公開 SDK ではないため backward compatibility shim は不要。

検証: ✅

- 402 の `accepts[0].maxAmountRequired` が `flovia.final_price` の atomic 値と一致（例 `"25000"` = `0.025`）。
- wallet-only と verified で `flovia.type` が変わる。

目的: current request pricing と post-payment next offer を分離する。

実装内容:

実装済み:

- ✅ Rules:

  - 2+ successful payments → `loyalty_bundle`
  - previous premium endpoint usage → `premium_continuation` は optional stretch のまま未実装。
- ✅ `flovia_next_offer` は current paid response amount / settled payment を変更しない。

検証: ✅

- 1回目 paid に `starter_upsell`、2回目以降に `loyalty_bundle`。
- dashboard revenue は settled payment amount だけから計算。

- Real x402 schema は placeholder と異なる可能性があるため `paymentMode` を明示する。
- Real Privy SDK 統合で `apps/web` の client/server 境界が変わる可能性。
- In-memory store は restart でデータ消失（Drizzle/Postgres phase まで許容）。
- `/v1/dev/*` simulation endpoints は本番でゲート / 削除が必要。
