# 002-xxx: Close MVP Gaps Toward Spec Compliance

## Goal

`001-xxx` で作った local demo flow を、最新の `docs/SPEC.md` に近づける。

優先する完成状態:

- `flovia` extension と Flovia API payload が MVP canonical shape に揃っている。
- wallet-only Privy user と verified Privy user の demo path が表現できる。
- merchant response が `unlockable_discount` と `verified_user_discount` を返せる。
- paid response に `flovia_next_offer` を返せる。
- frontend が dashboard API の real data を読める。
- real Privy / real x402 / Postgres に進む前の simulation contract が固まっている。

  - `starter_upsell`
  - `loyalty_bundle`
- `IdentityConfidenceSchema` を追加する。
  - `anonymous`
  - `wallet_only`
  - `verified_contact`

検証:

## Phase 2: Identity Simulation Model

目的: real Privy の前に、wallet-only と verified を backend が区別できるようにする。

実装内容:

- `OfferQuoteRequestSchema.metadata` を拡張する。
  - `has_agent_authorization?: boolean`
  - `identity_confidence?: IdentityConfidence`
  - `linked_account_types?: LinkedAccountType[]`
  - `privy_did?: string`
- 既存の `privy_authorized` は廃止または `has_agent_authorization` に置き換える。
- `packages/db` に simulation user store を追加する。
  - wallet address
  - optional privy DID
  - identity confidence
  - linked account types
  - agent authorization state
- `apps/api` に simulation endpoints を追加する。
  - `POST /v1/sim/users`
  - `POST /v1/sim/users/:wallet/link-account`
  - `POST /v1/sim/users/:wallet/authorize-agent`
  - `GET /v1/sim/users/:wallet`
- merchant quote flow は wallet から local simulation user を解決できるようにする。

Demo states:

- anonymous wallet: no Flovia user context
- wallet-only Privy user: `identity_confidence = wallet_only`, active authorization true
- verified Privy user: `identity_confidence = verified_contact` or `verified_social`, active authorization true
- strong auth user: `identity_confidence = strong_auth`, active authorization true

検証:

- wallet-only user quote returns `unlockable_discount` with final price equal base price.
- verified user quote returns `verified_user_discount` with final price 50% of base price.
- anonymous wallet quote returns base price and no unlock prompt.

## Phase 3: Offer Engine Alignment

目的: Pattern 1 を core path として正しく実装する。

実装内容:

- `computeOffer` を identity confidence based に変更する。
- Pricing rules:
  - anonymous: `type = base_price`, `policy = base_price`, `final_price = base_price`
  - wallet-only + authorized: `type = unlockable_discount`, `policy = base_price_until_verified`, `final_price = base_price`, `unlock.target_final_price = base_price * 0.5`
  - verified contact/social + authorized: `type = verified_user_discount`, `policy = verified_user_discount`, `final_price = base_price * 0.5`
  - strong auth + authorized: same price as verified user for MVP, with stronger reason code
- `reason_codes` を SPEC に合わせる。
  - `anonymous_wallet`
  - `low_identity_confidence`
  - `privy_agent_authorized`
  - `verified_privy_user`
  - `strong_auth_signal`
- repeat buyer / bundle logic は current-request pricing から外し、Phase 5 の `flovia_next_offer` に移す。

Money handling:

- JS `Number` arithmetic をやめる。
- MVP では decimal string を USDC 6 decimals の bigint atomic units に変換して計算する helper を `packages/shared` または `packages/config` に置く。
- helper examples:
  - `parseUsdc("0.05") -> 50000n`
  - `formatUsdc(25000n) -> "0.025"`
  - `multiplyBps(50000n, 5000) -> 25000n`

検証:

- `0.05 * 0.5` が必ず `0.025` になる。
- `0.01`, `0.025`, `0.05`, `0.08` の roundtrip tests を追加する。

## Phase 4: SDK And Merchant 402 Response

目的: merchant response を SPEC の canonical 402 に揃える。

実装内容:

- SDK の response shape を変更する。
  - `flovia.request_id`
  - `flovia.quote_id`
  - `flovia.buyer`
  - `flovia.offer`
- `accepts[0]` を mode によって分ける。
  - simulation mode: current simplified shape を維持してもよいが `simulation: true` を明示する。
  - real x402 mode: `maxAmountRequired`, asset address, payTo を使う。
- config に asset address placeholder を追加する。
  - `baseSepoliaUsdcAsset`
  - `baseUsdcAsset`
- SDK option に `paymentMode: "simulation" | "x402"` を追加する。
- `X-PAYMENT` parse は simulation 用として明示する。

Compatibility policy:

- 今は外部公開 SDK ではないため backward compatibility shim は不要。
- demo-agent と README は新 shape に合わせて同時に更新する。

検証:

- 402 response の `accepts[0].amount` または `maxAmountRequired` が `flovia.offer.final_price` と一致する。
- wallet-only と verified で 402 response の `flovia.offer.type` が変わる。

## Phase 5: Payment Event And `flovia_next_offer`

目的: current request pricing と post-payment next offer を分離する。

実装内容:

- `PaymentEventSchema` に `offer_selected` と `offer_type` を明確化する。
- `packages/db` の history aggregation を privy DID or wallet に紐づける。
- paid merchant response に `flovia_next_offer` を optional で付与する。
- Rules:
  - first successful payment: `starter_upsell`
  - 2+ successful payments: `loyalty_bundle`
  - previous premium endpoint usage: `premium_continuation` は optional stretch
- `flovia_next_offer` は current paid response amount を変更しない。

検証:

- 1回目の paid response に `starter_upsell` が出る。
- 2回目以降に `loyalty_bundle` が出る。
- dashboard の revenue は settled payment amount だけから計算される。

## Acceptance Criteria

この phase は以下を満たせば完了。

- `bun run typecheck` が通る。
- `bun run test` が通る。
- `bun run --filter='web' build` が通る。
- wallet-only simulation user receives HTTP 402 with `flovia.offer.type = "unlockable_discount"`.
- verified simulation user receives HTTP 402 with `flovia.offer.type = "verified_user_discount"` and `final_price = "0.025"` for base price `"0.05"`.
- anonymous wallet receives base price without Privy reason codes.
- demo-agent can parse `flovia.offer` and retry payment simulation.
- paid merchant response can include `flovia_next_offer` without changing current paid response amount.
- dashboard frontend reads API data instead of only showing static preview.

## Risks

- SPEC is still changing; avoid over-abstracting until canonical schema is stable.
- Real x402 schema may differ from placeholder examples; keep `paymentMode` explicit.
- Real Privy SDK integration may force client/server boundary changes in `apps/web`.
- In-memory store will lose data on restart; this is acceptable until Drizzle/Postgres phase.
